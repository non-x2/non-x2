# 📄 指示書：Git 操作手順書（マージまで）

> 「枝で作って、幹に取り込む」までの操作を**ステップで確認**できるマニュアルです。
> 用語の意味はスライド [`slides_git_claude入門.md`](slides_git_claude入門.md) を参照。
> ※ 操作はClaudeに頼めばやってくれます。本書は「何が起きているか」を理解するためのものです。

---

## 🗺 全体の流れ（地図）

```
① ブランチを作る   → 作業用の枝を生やす
② 変更してコミット → 区切りごとに記録
③ プッシュ         → GitHubへ送る
④ PRを作る         → 取り込みを提案（最初はドラフト）
⑤ Readyにする      → 完成宣言
⑥ マージ           → 本流(main)に合流 ✅
```

---

## ① ブランチを作る

**目的**：`main` を直接いじらず、安全に作業するため。

- Claudeへの頼み方：「`○○` 用の新しいブランチを作って作業して」
- コマンド例：
  ```bash
  git checkout -b claude/my-feature   # 枝を作って移動
  git branch --show-current           # 今いる枝を確認
  ```

> 💡 ブランチ名は内容が分かる名前に（例：`claude/git-guide-slides`）。

---

## ② 変更してコミット

**目的**：変更を「区切り」で記録する（セーブ）。

- Claudeへの頼み方：「変更できたらコミットして」
- コマンド例：
  ```bash
  git add .                            # 変更を記録対象に
  git commit -m "わかりやすい説明"     # 記録（セーブ）
  git log --oneline -5                 # 履歴を確認
  ```

> 💡 コミットメッセージは「何をしたか」が分かる一文に。

---

## ③ プッシュ（GitHubへ送る）

**目的**：ローカルの変更をGitHub（リモート）に保存・共有。

- Claudeへの頼み方：「GitHubにプッシュして」
- コマンド例：
  ```bash
  git push -u origin claude/my-feature
  ```

> 💡 これでGitHub上に枝が現れ、PRを作れるようになります。

---

## ④ プルリクエスト（PR）を作る

**目的**：「この変更を `main` に入れていい？」と提案する。

- Claudeへの頼み方：「PRをドラフトで作って」
- ポイント：
  - **ドラフト**＝まだ作業中の下書き状態
  - PRには「何を・なぜ変えたか」を書く（Claudeが用意します）
  - 差分（変更点）を見て確認できる

---

## ⑤ Ready for review にする

**目的**：「完成したのでマージできます」と宣言。

- Claudeへの頼み方：「PRをReadyにして」
- これをしないと（基本）マージできません。
- ドラフト⇔Readyは何度でも切り替え可能。

---

## ⑥ マージ（本流に合流）

**目的**：変更を `main` に取り込んで**正式採用**する。

- Claudeへの頼み方：「PRをマージして」
- マージ方式（迷ったら **squash**）：
  | 方式 | 特徴 |
  |------|------|
  | merge | 枝のコミットを全部残す |
  | **squash** | 1つにまとめて取り込む（履歴スッキリ）|
  | rebase | 直線的に並べ替えて取り込む |

- マージ後の後片付け：
  ```bash
  git checkout main          # 本流へ移動
  git pull origin main       # 最新を取り込む（マージ結果を反映）
  ```

---

## 🆘 困ったときのフレーズ

| 状況 | Claudeへの一言 |
|------|---------------|
| 今どうなってる？ | 「今のGitの状態を教えて」 |
| 変更を取り消したい | 「さっきの変更を元に戻して」 |
| マージしていいか不安 | 「マージ前に変更点を見せて」 |
| 用語が分からない | 「○○って何？」 |
| 間違えた気がする | 「直前の操作を確認して。やり直せる？」 |
