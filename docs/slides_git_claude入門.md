# 第1部：Git / GitHub の基礎 🧱

「枝で作って、幹に取り込む」が分かれば大丈夫

---

## そもそも Git とは？

**Git ＝ ファイルの変更履歴を記録する仕組み（バージョン管理）**

- いつ・誰が・どこを変えたかを全部記録
- 過去のどの時点にも戻れる
- 複数の作業を後から安全に合流できる

> 「セーブポイントを無限に作れて、いつでもロードできるRPG」とイメージするとラク。

---

## GitHub とは？

**GitHub ＝ Git の置き場所（クラウド）＋共同作業の場**

- 自分のPCの履歴を**ネット上に保存・共有**できる
- 変更の提案・レビュー・議論ができる
- 今回の `non-x2/non-x2` もGitHub上のリポジトリ

| 用語 | 意味 |
|------|------|
| リポジトリ(repo) | プロジェクトの保管場所 |
| ローカル | 自分のPC（今回はクラウドの作業環境）側 |
| リモート(origin) | GitHub側 |

---

## 4つの基本用語

| 用語 | ひとことで | たとえ |
|------|-----------|--------|
| **コミット** | 変更を1区切りで記録 | ゲームのセーブ |
| **ブランチ** | 作業用の枝分かれ | 下書き用のコピー |
| **プッシュ** | ローカル→GitHubへ送る | クラウドに保存 |
| **マージ** | 枝を本流に合流 | 下書きを本採用 |

---

## ブランチ：なぜ枝を分ける？

本流（`main`）を**直接いじらない**ための仕組み。

```
main ────●────────────────●  ← 本番（常に安全）
          \              /
作業枝 ─────●──●──●──────   ← ここで自由に試す
```

- `main` を壊さず、安心して試行錯誤できる
- 失敗しても枝を捨てればOK
- 完成したらマージで本流へ

---

## マージとは？（今回やったこと）

**マージ ＝ 作業ブランチの変更を、本流(`main`)に合流させて確定すること**

```
main ──●──●──────────────●(マージ後)
          \            /
作業枝 ────●──●(完成)──
```

- 「枝で作ったもの」を「幹」に取り込む
- これで変更が**正式採用**される

---

## プルリクエスト（PR）とは？

**PR ＝「この変更を main に入れていい？」という提案・確認の場**

1. 作業枝をプッシュ
2. **PRを作成**（差分を見せて提案）
3. レビュー・確認
4. **マージ**で取り込み完了 ✅

> PRがあると「入れる前に中身を確認」できる。チームでも個人でも安全。

---

## ドラフトPRとは？

**ドラフト ＝「まだ作業中／確認中」の下書き状態のPR**

- ドラフトのままだと（基本）マージできない
- 完成したら **Ready for review** に切り替え → マージ可能に

今回の流れ：
`ドラフト作成` → `中身を修正` → `Readyに変更` → `マージ` 🎉

---

## マージ方式の違い（軽く）

| 方式 | 何をする | 履歴 |
|------|---------|------|
| **merge** | そのまま合流 | 枝のコミットが全部残る |
| **squash** | 1つにまとめて合流 | スッキリ1コミット ← 今回これ |
| **rebase** | 並べ替えて合流 | 直線的になる |

> 迷ったら **squash** が初心者にはおすすめ（履歴が読みやすい）。

# 第2部：Claude への指示のコツ 🗣️

「良い指示」が「良い成果物」を作る

---

## 大原則：背景・目的・完成形を伝える

❌ 悪い例：「動画の資料作って」
✅ 良い例：
> 「**初心者の自分向けに**、Gitのマージの**使い方をスライドで**作って。
>  **専門用語はかみ砕いて**、**今回の実例も入れて**ほしい」

- **誰向け**か（自分／視聴者／中年家庭層 など）
- **何のため**か（学習／公開用 など）
- **どんな形**か（スライド／表／チェックリスト）

---

## コツ① まず「相談」、次に「実行」

- 大きめの作業は、いきなり作らず**方針を相談**するとズレが減る
- Claudeは作る前に**質問してくる**ことがある → 答えると精度UP
- 「**まず案を出して**」「**作る前に確認して**」と言うのも有効

> 今回も最初に「ジャンルは？トーンは？」と確認 → 手戻りゼロに。

---

## コツ② 具体的に、でも任せる所は任せる

- **譲れない条件は具体的に**（例：「ポイントは必ず3つ」「免責を入れて」）
- 細部は「**いい感じに**」「**おすすめで**」と任せてOK
- 例が一番強い：「**こういう感じで**」と見本を見せる

---

## コツ③ 少しずつ・確認しながら

- 一度に全部より、**段階的**に頼むと軌道修正しやすい
- 「ここをこう直して」と**部分修正**を重ねるのが効率的
- 画像・スクショを貼るのも有効（今回チャンネル情報を画像で共有）

---

## コツ④ 安全に関わる指示は明確に

- **公開・送信・削除・マージ**など“戻しにくい操作”は
  「やっていい」「まだ待って」をはっきり伝える
- Claudeは重要操作の前に**確認を取る**
- 「**勝手にマージしないで**」「**プッシュまででOK**」なども有効

---

## よく使う指示フレーズ集 🧰

- 「**まず方針を3案出して**、選ばせて」
- 「**専門用語はかみ砕いて**説明して」
- 「**初心者向けに**、たとえ話を入れて」
- 「**作る前に確認**して」「**部分的に直して**」
- 「**コミットして、PRはドラフトで**」
- 「**この画像の内容を反映して**」

# 第3部：今回の実例 📦

YouTubeテンプレを作って main にマージするまで

---

## やったこと（全体像）

1. **相談**：チャンネルのジャンル・トーン・成果物を確認
2. **作成**：`youtube/` にテンプレ群（企画〜公開チェック）を作成
3. **修正**：チャンネル情報の画像を反映し、実態に合わせて改訂
4. **プッシュ → PR**（ドラフト）作成
5. **Ready に変更 → squash マージ** で `main` に反映 ✅

---

## 実例で見る用語の対応

| 場面 | 使った用語 |
|------|-----------|
| 作業用の枝を作った | ブランチ `claude/youtube-...` |
| 変更を記録した | コミット（2回） |
| GitHubへ送った | プッシュ |
| 取り込み提案した | プルリクエスト #1 |
| 下書き状態だった | ドラフト |
| 本流に合流させた | マージ（squash）|

---

## ポイント：画像での共有が効いた

- チャンネルのAbout情報を**スクショで共有**
- → ジャンル・3ポイント・VOICEVOX・免責などが**一発で正確に**伝わった

> 文字で説明しづらい時は「**見せる**」が最強。

---

## まとめ 🎯

- **マージ ＝ 枝で作ったものを幹に取り込んで確定**
- ブランチ→コミット→プッシュ→PR→マージ の流れを押さえる
- Claudeには **背景・目的・完成形** を伝える
- **相談→具体指示→段階修正→安全確認** が成功のコツ

---
